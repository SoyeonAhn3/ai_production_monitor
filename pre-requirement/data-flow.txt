================================================================================
  AI Production Monitor - 데이터 흐름 정리
================================================================================

1. 전체 데이터 흐름 개요
================================================================================

  line_master (기준값 SSOT)
      |
      | 참조
      v
  generate_data_bank.js --> data_bank (120행, 이상 시나리오 5개 포함)
                                |
                                | 워크플로 A가 시간별로 읽음
                                v
                          production_week (시뮬레이터가 시간별 12행씩 append)
                                |
                                | 워크플로 B가 매시간 읽음
                                v
                          +---------------------+
                          | production_results   |
                          |---------------------|
                          | hourly_summary      | --> Power BI 대시보드
                          | anomaly_log         | --> Claude API / 이메일 / 워크플로 D
                          | daily_summary       | --> Power BI 대시보드
                          | error_log           | --> 디버깅 / 운영 추적
                          +---------------------+


2. Google Sheets 구성 (4개 시트)
================================================================================

  +------------------+-------------------+--------------------------------------+
  | Google Sheets    | 탭(시트)          | 용도                                 |
  +------------------+-------------------+--------------------------------------+
  | line_master      | line_master       | 라인/팀 구성 SSOT (12라인, 3팀)      |
  | data_bank        | data_bank         | 데모용 사전 데이터 (120행)           |
  | production_week  | production_week   | 이번 주 raw data (최대 720행)        |
  | production_results| hourly_summary   | 팀별 시간 집계                       |
  |                  | anomaly_log       | 이상 탐지 결과                       |
  |                  | daily_summary     | 일일 요약                            |
  |                  | error_log         | 에러 기록                            |
  +------------------+-------------------+--------------------------------------+


3. 데이터 생성 스크립트 (Phase 1)
================================================================================

  3-1. generate_line_master.js --> line_master 시트
  -----------------------------------------------
  - 생성 내용: 12라인 x 3팀 마스터 데이터
  - 컬럼: 라인ID, 라인명, 팀, 상위라인ID, 계층, 품목, 시간당목표, 정상불량률
  - 사용처:
    (1) 시스템 전체의 Single Source of Truth
    (2) 워크플로 B: 매 실행마다 읽어 라인 수, 기준값(시간당목표, 정상불량률) 동적 파악
    (3) data_bank 생성: target, defectRate를 기반으로 정상 범위 데이터 생성
    (4) Power BI: 라인/팀 구성 정보 조회
    (5) 워크플로 D: AI Agent가 라인 정보 조회 시 참조

  3-2. generate_data_bank.js --> data_bank 시트
  -----------------------------------------------
  - 생성 내용: 120행 (12라인 x 10시간) + 이상 시나리오 5개 오버라이드
  - 컬럼: 25개 (식별 7 + 시간당 5 + 누적 5 + 진척 3 + 가동 4 + 기타 1)
  - 이상 시나리오:
    (1) CNC 1호기(L03)   11:00~14:00  설비 고장 (급감 -> 정지 -> 시운전)
    (2) 도장 라인(L07)   09:00~12:00  불량률 정상의 8배
    (3) 사출 1호기(L08)  13:00~15:00  금형 이상 (급감 -> 심화 -> 복구중)
    (4) 조립 2라인(L11)  10:00~12:00  작업자 교체 (생산 40~55% + 불량 5배)
    (5) 프레스 1호기(L01) 15:00~17:00 자재 대기 (가동률 급락)
  - 사용처:
    (1) 워크플로 A(시뮬레이터) 전용 소스 데이터
    (2) 이상 탐지 로직 검증용 의도된 이상 데이터 저장소

  3-3. generate_empty_files.js --> production_week + production_results 시트
  -----------------------------------------------
  - 생성 내용: 헤더만 생성 (데이터 없이 컬럼 구조 초기화)
  - 대상:
    (1) production_week     (25개 컬럼) - 시뮬레이터가 쌓는 이번 주 raw data
    (2) hourly_summary      (11개 컬럼) - 워크플로 B가 매시간 쓰는 팀별 시간 집계
    (3) anomaly_log         (16개 컬럼) - 이상 탐지 결과 + AI 해석 + 알림 상태
    (4) daily_summary       (15개 컬럼) - 워크플로 C가 주간 백업 시 쓰는 일일 요약
    (5) error_log           (8개 컬럼)  - 워크플로 실행 중 에러 기록


4. 워크플로별 데이터 흐름
================================================================================

  4-1. 워크플로 A (테스트 시뮬레이터)
  -----------------------------------------------
  트리거: Webhook 수동 클릭
  흐름:
    Webhook 클릭
      --> Static Data에서 카운터 읽기 (0~9)
      --> data_bank에서 해당 시간의 12행 필터링
      --> production_week에 12행 append
      --> 카운터 +1
      --> 브라우저에 결과 반환

  읽는 데이터: data_bank
  쓰는 데이터: production_week
  리셋: /webhook/reset --> 카운터 초기화 + production_week 재생성

  4-2. 워크플로 B (모니터 - 이상 탐지)
  -----------------------------------------------
  트리거: Schedule Trigger 매시간 10분
  흐름:
    production_week 읽기
      --> line_master 읽기
      --> 데이터 검증 (null/빈칸/라인 누락)
      --> rules.json 기반 8룰 이상 탐지 + hourly_summary 생성
      --> hourly_summary 탭에 저장
      --> anomaly_log 최근 7일 읽기
      --> 패턴 분류 (신규/반복/악화)
      --> anomaly_log에 이상 저장
      --> [심각/악화] Claude API 호출 --> AI 해석 업데이트 --> Gmail 알림
      --> [중간만]   Gmail 알림 (AI 없음)
      --> [낮음만]   로그만 저장

  읽는 데이터: production_week, line_master, anomaly_log (과거 7일)
  쓰는 데이터: hourly_summary, anomaly_log, error_log

  4-3. 워크플로 C (주간 백업)
  -----------------------------------------------
  트리거: Cron 매주 월요일 07:50
  흐름:
    production_week 읽기
      --> 날짜 범위 계산 (지난주 월~일)
      --> daily_summary 집계 (팀별 일별 요약)
      --> daily_summary 탭에 저장
      --> production_week 클리어 (헤더만 남김)

  읽는 데이터: production_week
  쓰는 데이터: daily_summary, production_week (클리어)

  4-4. 워크플로 D (AI Agent 일일 리포트)
  -----------------------------------------------
  트리거: Cron 매일 08:00
  흐름:
    anomaly_log, daily_summary, production_week, line_master 읽기
      --> Claude API Agent에 데이터 + 도구 4개 제공
      --> Agent가 자율적으로 도구 호출 (최대 8회)
          - get_anomaly_log(날짜, 심각도, 라인)
          - get_daily_summary(날짜, 팀)
          - get_line_master()
          - get_hourly_detail(날짜, 라인)
      --> 리포트 생성 (심각도에 따라 간단/보통/상세)
      --> Gmail 발송 (상세 시 경영진 CC)

  읽는 데이터: anomaly_log, daily_summary, production_week, line_master
  쓰는 데이터: 없음 (이메일 발송만)


5. 외부 연결 데이터 흐름
================================================================================

  5-1. Claude API (Anthropic)
  -----------------------------------------------
  - 워크플로 B: 심각/악화 이상 건에 대해 원인 분석 요청
    입력: 마스킹된 이상 데이터 (LINE_A, PROD_01 등)
    출력: 원인 분석 + 패턴 검증 --> anomaly_log의 ai_insight 컬럼에 저장
  - 워크플로 D: Agent Tool Use로 자율 데이터 조회 + 리포트 생성
    입력: 도구 4개 + 시스템 프롬프트
    출력: JSON 형식 일일 리포트

  5-2. Gmail
  -----------------------------------------------
  - 워크플로 B: 즉시 알림 (심각+중간: 즉시 / 낮음: 로그만)
  - 워크플로 D: 일일 리포트 발송 (07:40 예정)
  - 중복 방지: n8n Static Data에서 {라인ID}_{이상유형} 키로 추적

  5-3. Power BI
  -----------------------------------------------
  - Google Sheets "Publish to web" --> CSV 링크 생성
  - Power BI Web connector로 CSV 링크 연결
  - 데이터 소스:
    (1) hourly_summary --> 시간별 팀 집계 차트
    (2) anomaly_log    --> 이상 탐지 현황 + AI 해석
    (3) daily_summary  --> 일별 추이 차트
    (4) line_master    --> 라인/팀 필터 구성
  - 자동 갱신: 8회/일 (Power BI Gateway)


6. 컬럼 상세 구조
================================================================================

  6-1. production_week / data_bank (25개 컬럼)
  -----------------------------------------------
  [식별 7개]
    날짜, 시간, 라인ID, 라인명, 팀, 생산품목, 일일목표

  [시간당 5개]
    시간당생산, 시간당양품, 시간당불량, 시간당폐기, 시간당불량률(%)

  [누적 5개]
    누적생산, 누적양품, 누적불량, 누적폐기, 누적불량률(%)

  [진척 3개]
    달성률(%), 예상달성률(%), 달성갭(%p)

  [가동 4개]
    시간당가동(분), 시간당비가동(분), 시간당가동률(%), 누적가동률(%)

  [기타 1개]
    이상플래그

  6-2. anomaly_log (16개 컬럼)
  -----------------------------------------------
    날짜, 시간, 라인ID, 라인명, 팀, 품목,
    type, severity, detail,
    pattern_type, recurrence_count, ai_insight, ai_parsed,
    run_id, idempotency_key, notification_status

  6-3. hourly_summary (11개 컬럼)
  -----------------------------------------------
    날짜, 시간, 팀, 작업건수, 계획합계, 실적합계, 달성률(%),
    불량합계, 폐기합계, 평균가동률(%), 비가동합계(분)

  6-4. daily_summary (15개 컬럼)
  -----------------------------------------------
    날짜, 팀, 총계획, 총실적, 일일달성률(%),
    총불량, 총폐기, 일일불량률(%),
    평균가동률(%), 총비가동시간(분),
    이상건수_심각, 이상건수_중간, 이상건수_낮음,
    반복이상건수, 악화이상건수

  6-5. error_log (8개 컬럼)
  -----------------------------------------------
    날짜, 시간, run_id, workflow, step, error_type, error_message, action_taken

  6-6. line_master (8개 컬럼)
  -----------------------------------------------
    라인ID, 라인명, 팀, 상위라인ID, 계층, 품목, 시간당목표, 정상불량률


7. 데이터 무결성 보장
================================================================================

  7-1. 중복 방지
  -----------------------------------------------
  - idempotency_key: {라인ID}_{룰ID}_{날짜}_{시간}
    동일 키가 이미 있으면 저장하지 않음 (재실행 시 중복 방지)
  - run_id: B_{날짜}_{시간}
    한 번의 워크플로 실행에서 생성된 모든 레코드 추적

  7-2. 알림 중복 방지
  -----------------------------------------------
  - 심각: 60분 후 미해결 시 재알림
  - 중간: 사이클당 1회
  - 낮음: 이메일 없음 (로그만, 일일 리포트에 포함)
  - notification_status: pending --> sent / failed / skipped

  7-3. 에러 처리 (7개 에러 포인트)
  -----------------------------------------------
  (1) Google Sheets 읽기 실패   --> error_log 기록 --> 종료 (다음 사이클 재시도)
  (2) 데이터 검증 실패          --> 해당 라인 제외 --> 유효 라인만 계속
  (3) 탐지 계산 에러            --> 해당 룰만 스킵 --> 나머지 룰 계속
  (4) Google Sheets 쓰기 실패   --> 2회 재시도(10초) --> 이메일은 발송 시도
  (5) Claude API 실패           --> 1회 재시도(15초) --> 폴백 템플릿 메시지
  (6) AI 응답 파싱 실패         --> raw text 저장, ai_parsed=false
  (7) Gmail 발송 실패           --> 2회 재시도(10초) --> notification_status="failed"
