1. 프로젝트 개요
================================================================================

[프로젝트명]
AI BI Assistant — 생산 실적 이상 탐지 및 AI 자동 해석 시스템

[프로젝트 성격: 포트폴리오 데모]
이 프로젝트는 포트폴리오/데모 목적으로 제작됩니다.
실제 제조 현장에 적용하려면 인증 체계, 에러 핸들링, 로깅,
멀티 사용자 지원, 권한 관리 등 추가 레이어가 필요합니다.
(14장 "실제 현장 적용 가이드"에서 상세 설명)

다만, 핵심 로직(이상 탐지, AI 해석, 알림, 대시보드)은
실무에서도 동일하게 사용할 수 있는 수준으로 설계합니다.

[핵심 컨셉]
생산 현장의 실적 데이터를 1시간 간격으로 자동 모니터링하고,
이상이 감지되면 AI가 원인을 분석하여 담당자에게 이메일로 즉시 알림을 보내고,
Power BI 대시보드에서 상세 분석을 확인할 수 있는 완전 자동화 시스템.
n8n 셀프호스팅(Docker)으로 24시간 운영.

[해결하는 문제]
- 생산 이상(생산량 급감, 불량률 급등, 가동률 저하)을
  사람이 수동으로 발견하기 전에 AI가 자동 탐지
- Power BI 대시보드를 만들어놓아도 경영진이 안 보는 문제
  → 이메일로 핵심 인사이트를 먼저 전달
- 보고서 작성에 소요되는 반복 업무 자동화
- 회사 데이터를 AI에 넘기는 보안 우려
  → 마스킹 처리 + 템플릿 모드로 해결

[대상 사용자]
- 생산 관리자: 이메일 알림으로 즉시 이상 파악, 필요 시 Power BI에서 상세 확인
- 경영진: 주간/월간 AI 인사이트 리포트로 생산 현황 파악
- 품질 담당자: 불량률 추이 모니터링 및 조기 경고

2. 시스템 아키텍처 (v2.0 변경)
================================================================================

[전체 파이프라인 흐름]
Google Sheets (원본 데이터 입력 / 시뮬레이터)
  → n8n 셀프호스팅 Docker (60분 스케줄 / 테스트 시 Webhook)
  → Python 이상 탐지 (n8n Code 노드)
  → Claude API (AI 해석, 마스킹된 집계 데이터만 전달)
  → Gmail 알림 (이상 발견 시 즉시)
  → SharePoint Excel 결과 저장 (← v2.0 변경: Google Sheets → SharePoint)
  → Power BI Service 대시보드 (자동 새로고침 하루 8회)

[v1.0 대비 주요 변경]
1. n8n Cloud → 셀프호스팅(Docker)
   - 이유: n8n Cloud Code 노드는 JavaScript만 지원.
     Python(Prophet, scikit-learn 등)을 쓰려면 셀프호스팅 필수.
   - 장점: 무료, 무제한 실행, Python 지원, 라이브러리 자유 설치
   - 단점: Docker 환경 필요 (개인 PC 또는 클라우드 VM)
   - 대안 검토:
     a) JS로 이상 탐지 작성 → 룰 기반은 가능하나 ML 모델 불가
     b) 외부 Python API 서버 분리 → 서버 추가 관리 부담
     → 셀프호스팅이 가장 깔끔

2. 분석 결과 저장소: Google Sheets → SharePoint Excel
   - 이유: Power BI Service에서 Google Sheets 직접 연결은 공식 미지원.
     Web 커넥터 + CSV 공개 링크는 보안 문제.
   - 변경: n8n이 분석 결과(hourly_summary, anomaly_log)를
     SharePoint/OneDrive의 Excel 파일에 저장
   - Power BI가 SharePoint Excel을 공식 지원하므로 연결 안정적
   - E3 라이선스에 SharePoint 포함 → 추가 비용 없음
   - 원본 데이터 입력(data_bank, raw_current)은 Google Sheets 유지

3. Google Sheets API 최적화
   - 개별 API 호출 대신 batch update(spreadsheets.values.batchGet,
     spreadsheets.values.batchUpdate) 사용
   - 한 사이클 API 호출: 5~6회 → 2~3회로 감소
   - 분당 60 요청 제한에 여유 확보

[핵심 설계 원칙]
1. 하이브리드 클라우드 구조
   - n8n: 셀프호스팅 Docker (PC 또는 VM에서 24시간 실행)
   - 데이터: Google Sheets(입력) + SharePoint Excel(출력)
   - 대시보드: Power BI Service(웹)
   - ※ n8n이 로컬이므로 "PC 없이 24시간"은 불가.
     상시 운영하려면 클라우드 VM(AWS Free Tier 등) 또는 항시 켜진 PC 필요.

2. 변경 시에만 분석 실행
   - raw_current와 previous 비교 → 동일하면 아무것도 안 함
   - API 비용 + Google Sheets 호출 횟수 절약

3. 실패 안전(Fail-safe) 설계 (← v2.0 추가)
   - previous 복사를 파이프라인 맨 마지막에 수행
   - 중간 실패 시: 다음 사이클에서 재감지 (누락보다 중복이 안전)
   - last_processed_timestamp 플래그로 이중 확인

4. 중복 알림 방지 (← v2.0 상태 저장 명시)
   - 상태 저장: n8n Static Data에 JSON으로 마지막 알림 시각 기록
     예: {"L03_설비고장": "2026-03-23T11:00:00", "L07_불량급등": "..."}
   - 처음 감지: 즉시 알림
   - 30분 이내 같은 유형: 알림 억제
   - 30분 후에도 지속: 에스컬레이션 알림

6. 이상 탐지 방법론
================================================================================

[탐지 기준 8가지]

1. 생산량 급감 (심각도: 높음)
   - 조건: 시간당생산 < 이전시간 대비 40% 이상 감소
   - 의미: 설비 고장, 자재 부족 등 즉시 대응 필요

2. 가동률 저하 (심각도: 높음)
   - 조건: 시간당가동률 < 50%
   - 의미: 30분 이상 비가동, 설비 점검 필요

3. 불량률 급등 (심각도: 높음)
   - 조건: 시간당불량률 > 해당 라인 정상불량률의 3배
   - 의미: 원자재 문제, 설비 세팅 이상 등

4. 달성률 지연 (심각도: 중간)
   - 조건: 달성갭 < -15%p
   - 의미: 이 속도면 일일 목표 미달 위험

5. 생산 정체 (심각도: 높음)
   - 조건: 2시간 연속 시간당생산 = 0
   - 의미: 라인 완전 정지 상태

6. 폐기 발생 (심각도: 중간)
   - 조건: 시간당폐기 > 0 (해당 라인 정상 폐기율 초과 시)
   - 의미: 품질 이슈 모니터링

7. 복합 이상 (심각도: 높음)
   - 조건: 생산량 감소 + 불량률 상승 동시 발생
   - 의미: 작업자 교체, 설비 이상 등 복합 원인

8. 가동률 급락 (심각도: 높음)
   - 조건: 이전시간 대비 가동률 30%p 이상 하락
   - 의미: 갑작스러운 비가동 발생

[탐지 구조: Config-driven Rules] (← v2.1 추가)

이상 탐지 룰을 Python 코드에 하드코딩하지 않고,
JSON 설정 파일(rules.json)과 범용 엔진(engine.py)으로 분리한다.

  - rules.json: 어떤 컬럼을, 어떤 방식으로, 어떤 기준값과 비교할지 정의
  - engine.py: rules.json을 읽어서 자동 실행하는 범용 코드 (한 번 작성 후 수정 불필요)

  새 데이터가 들어오면: rules.json에 룰만 추가 → 코드 수정 없음
  새 컬럼이 추가되면: rules.json에 target만 변경 → 코드 수정 없음

  지원 비교 타입 (고정 7가지):
    - drop_rate: 이전 대비 N% 감소 (예: 생산량 40% 급감)
    - exceeds_baseline: 기준값의 N배 초과 (예: 불량률이 정상의 3배)
    - below_threshold: 고정값 미만 (예: 가동률 50% 미만)
    - above_threshold: 고정값 초과 (예: 폐기 0 초과)
    - consecutive_zero: 연속 N회 0 (예: 2시간 연속 생산 0)
    - drop_points: 이전 대비 N%p 하락 (예: 가동률 30%p 급락)
    - compound: 두 조건 동시 충족 (예: 생산 감소 + 불량 상승)

  ※ 실무 확장 시 유의사항:
    위 7가지 비교 타입은 제조 생산 데이터 기준으로 설계된 개발/데모용입니다.
    실무에서 다른 도메인(물류, 에너지, 금융 등)의 데이터를 적용할 경우,
    기존 7가지 타입으로 커버되지 않는 비교 로직이 필요할 수 있으며
    이 경우 engine.py에 새 type을 추가하는 코드 수정이 필요합니다.
    (예: 이동평균 비교, 시간대별 기대값 비교, 다변량 상관관계 비교 등)

[탐지 방법 단계]
MVP 1주차: Config-driven 룰 기반 임계치 (위 8가지 룰 → rules.json)
MVP 2주차: 이전 시간 대비 통계적 편차 탐지

[이 프로젝트의 정직한 포지셔닝]
이 프로젝트는 포트폴리오/데모 목적입니다.
핵심 로직(이상 탐지 알고리즘, AI 해석, 알림 구조, 대시보드)은
실무에서 동일하게 사용할 수 있지만,
실제 현장 적용에는 아래의 추가 레이어가 필요합니다.

[포트폴리오와 동일한 부분 — 변경 없이 재사용 가능]
  ✓ 이상 탐지 로직 (8가지 기준 + Python 코드)
  ✓ AI 해석 생성 (Claude API + 마스킹 처리)
  ✓ 중복 알림 방지 로직
  ✓ 이메일 알림 구성 (HTML 템플릿)
  ✓ Power BI 대시보드 구조 (4개 페이지)
  ✓ 데이터 수명주기 관리 전략

[데이터 소스만 교체하면 되는 부분]
  데이터 입력:    Google Sheets 시뮬레이터 → MES/ERP → SharePoint
  데이터 읽기:    n8n Google Sheets 노드 → n8n SharePoint/DB 노드
  알림:          Gmail → Outlook/Teams (E3 포함)
  결과 저장:      이미 SharePoint (변경 불필요)

[실무에서 추가로 필요한 레이어 — 이 프로젝트에 포함되지 않음]

  1. 인증 및 권한 관리
     - n8n 접근 권한 (관리자 vs 뷰어)
     - SharePoint 파일 접근 권한 설정
     - Power BI 대시보드 공유 범위 설정
     - API 키 관리 (Claude API, n8n Credentials)

  2. 에러 핸들링 및 로깅
     - n8n 워크플로 실패 시 관리자 알림
     - API 호출 실패 시 재시도 로직 (exponential backoff)
     - 실행 이력 로깅 (성공/실패/스킵 기록)
     - 데이터 정합성 체크 (행 수, 컬럼 형식 검증)

  3. 멀티 사용자 / 멀티 공장
     - 공장별/라인그룹별 알림 수신자 분리
     - 사용자별 대시보드 필터
     - 여러 공장의 데이터를 하나의 파이프라인에서 처리

  4. 네트워크 보안
     - VPN 내에서 n8n 접근 제한
     - API 통신 암호화
     - 데이터 암호화 at rest

  5. 운영 안정성
     - n8n Docker 컨테이너 자동 재시작
     - 디스크 용량 모니터링
     - 백업/복구 절차
     - SLA 정의 (알림 지연 허용 범위 등)

  6. 스케일링
     - 라인 수 증가 시 데이터 처리 성능
     - Google Sheets → DB 전환 시점 판단
     - n8n 워크플로 실행 시간 모니터링
