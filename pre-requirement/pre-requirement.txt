1. 프로젝트 개요
================================================================================

[프로젝트명]
AI BI Assistant — 생산 실적 이상 탐지 및 AI 자동 해석 시스템

[프로젝트 성격]
포트폴리오/데모 목적으로 제작.
핵심 로직(이상 탐지, AI 해석, 알림, 대시보드)은 실무 수준으로 설계하되,
인증·에러 핸들링·멀티 사용자 등 운영 레이어는 포함하지 않음.
실제 현장 적용에 필요한 추가 레이어는 15장에서 별도 정리.

[핵심 컨셉]
생산 현장의 실적 데이터를 1시간 간격으로 자동 모니터링하고,
이상이 감지되면 AI가 원인을 분석하여 담당자에게 이메일로 즉시 알림을 보내고,
Power BI 대시보드에서 상세 분석을 확인할 수 있는 자동화 시스템.

[해결하는 문제]
- 생산 이상을 사람이 수동으로 발견하기 전에 자동 탐지
- 대시보드를 안 보는 문제 → 이메일로 핵심 인사이트 먼저 전달
- 보고서 작성 반복 업무 자동화
- 회사 데이터 보안 우려 → 마스킹 처리 + 템플릿 모드

[대상 사용자]
- 생산 관리자: 이메일 알림으로 즉시 이상 파악, Power BI에서 상세 확인
- 경영진: 일일/주간 AI 인사이트 리포트
- 품질 담당자: 불량률 추이 모니터링 및 조기 경고

2. 시스템 아키텍처
================================================================================

[전체 파이프라인]
SharePoint Excel (주간 raw data, 외부에서 시간별 행 추가)
  → n8n 셀프호스팅 Docker (60분 스케줄)
  → Python 이상 탐지 (Config-driven, n8n Code 노드)
  → Python 패턴 1차 분류 (규칙 기반, 로컬)
  → Claude API (심각/악화 건만, 마스킹 데이터)
  → Gmail 알림 (심각+중간: 즉시 / 전체: 다음날 08:00 일일 리포트)
  → SharePoint Excel 결과 저장
  → Power BI Service 대시보드 (SharePoint 직접 연결)

[설계 원칙과 선택 이유]

1. SharePoint 단일 저장소
   - 입력, 출력, 백업, 설정 전부 SharePoint에 통일
   - 이유: Power BI Service가 SharePoint Excel을 공식 지원하므로 연결이 안정적.
     Google Sheets는 Power BI와 직접 연결이 공식 미지원이라
     CSV 공개 링크 등 우회가 필요한데, 이는 보안 문제가 있음.
   - M365 E3에 SharePoint 포함 → 추가 비용 없음
   - 실무 적용 시: 동일하게 SharePoint 사용 (변경 불필요)

2. n8n 셀프호스팅 (Docker)
   - 이유: n8n Cloud의 Code 노드는 JavaScript만 지원하여
     셀프호스팅하면 Python Code 노드 사용 가능.
   - 단점: Docker가 돌아갈 PC 또는 VM이 항시 켜져 있어야 함.
     데모 시에는 로컬 PC에서, 상시 운영이 필요하면 클라우드 VM(AWS Free Tier 등) 활용.
   - 실무 적용 시: 사내 서버 또는 클라우드 VM에서 Docker로 운영

3. 변경 시에만 분석 실행
   - 주간 파일의 마지막 시간과 직전 시간을 비교 → 동일하면 스킵
   - 이유: 데이터가 안 바뀌었는데 매번 분석/AI 호출하면 비용 낭비.
     Google Sheets API에는 분당 60회 제한도 있었으므로 호출 최소화가 중요.
   - 파일 자체가 없으면(새 주 시작, 아직 데이터 미도착) 스킵

4. Config-driven 설계
   - 이상 탐지 룰: rules.json
   - 라인/팀 구성: line_master.xlsx
   - 이유: 기준값이나 라인 수가 바뀔 때마다 Python 코드를 수정하면
     유지보수가 어렵고 실수가 발생함. 설정 파일만 수정하면 코드 변경 없이
     즉시 반영되도록 분리. 실무에서 흔히 쓰이는 패턴.

5. 중복 알림 방지
   - n8n Static Data에 마지막 알림 시각을 JSON으로 기록
   - 심각만 30분 반복 (미해결 시 에스컬레이션)
   - 중간: 사이클당 1회만 발송
   - 낮음: 이메일 없음 (로그 + 일일 리포트에 포함)
   - 이유: 같은 문제로 매시간 이메일이 오면 관리자가 무시하게 됨.
     "심각"은 즉각 대응이 필요하므로 30분마다 반복하되,
     "중간"은 인지만 시키면 되므로 1회, "낮음"은 일일 리포트로 충분.

6. Schedule Trigger 
  : 매시간 10분 (08:10, 09:10, 10:10...)으로 실행. 정각에 외부 시스템이 데이터를 append하므로 10분 오프셋을 두어 반쪽 데이터 읽기를 방지. 
    추가로 데이터 검증에서 이번 시간 라인 수가 line_master의 라인 수와 다르면 데이터 불완전으로 판단하여 스킵.

4. 데이터 구조
================================================================================

[설계 원칙: 데모 데이터와 시스템 설계를 분리]

  데모에서는 12라인 3팀으로 고정되어 있지만, 시스템 코드에는 "12"라는 숫자가
  하드코딩되지 않음. 모든 라인/팀 정보는 line_master 설정 테이블에서 읽으므로,
  라인 추가/삭제 시 코드 수정 없이 테이블만 변경하면 됨.

  ┌─────────────┬──────────────┬─────────────────────────┐
  │    항목     │  데모 (샘플) │      시스템 설계         │
  ├─────────────┼──────────────┼─────────────────────────┤
  │ 라인 수     │ 12개         │ line_master에서 동적(N개)│
  ├─────────────┼──────────────┼─────────────────────────┤
  │ 팀 수       │ 3개          │ line_master에서 동적(N개)│
  ├─────────────┼──────────────┼─────────────────────────┤
  │ 하위 작업반 │ 없음         │ 스키마 지원 (아래 설명)  │
  ├─────────────┼──────────────┼─────────────────────────┤
  │ 운영 시간   │ 08:00~17:00  │ 실무 시 설정 가능       │
  ├─────────────┼──────────────┼─────────────────────────┤
  │ 운영 일수   │ 월~토 (6일)  │ 실무 시 설정 가능       │
  └─────────────┴──────────────┴─────────────────────────┘


[line_master 설정 테이블]

  파일: line_master.xlsx (SharePoint /config/)
  역할: 라인/팀 구성의 단일 진실 소스 (Single Source of Truth)
        engine.py가 매 실행마다 이 테이블을 읽어서
        라인 수, 팀 수, 라인별 기준값을 동적으로 파악함

  컬럼:
    라인ID | 라인명 | 팀 | 상위라인ID | 계층 | 품목 | 시간당목표 | 정상불량률

  데모 데이터:
    L01 | 프레스 1호기  | 1팀 | null | line | 브라켓 A  | 120 | 0.8%
    L02 | 프레스 2호기  | 1팀 | null | line | 브라켓 B  | 100 | 1.0%
    L03 | CNC 1호기    | 1팀 | null | line | 샤프트 A  | 45  | 1.5%
    L04 | CNC 2호기    | 1팀 | null | line | 샤프트 B  | 45  | 1.5%
    L05 | 용접 1호기   | 2팀 | null | line | 프레임 A  | 30  | 2.0%
    L06 | 용접 2호기   | 2팀 | null | line | 프레임 B  | 30  | 2.0%
    L07 | 도장 라인    | 2팀 | null | line | 커버 A    | 80  | 3.0%
    L08 | 사출 1호기   | 3팀 | null | line | 하우징 A  | 200 | 1.2%
    L09 | 사출 2호기   | 3팀 | null | line | 하우징 B  | 200 | 1.2%
    L10 | 조립 1라인   | 3팀 | null | line | 모듈 A    | 60  | 0.5%
    L11 | 조립 2라인   | 3팀 | null | line | 모듈 B    | 60  | 0.5%
    L12 | 검사/포장    | 3팀 | null | line | 완제품 A  | 150 | 0.3%

  [상위라인ID와 계층 컬럼 설명]

    실제 제조 현장에서는 하나의 라인 안에 여러 하위 공정(작업반)이 있을 수 있음.
    예: CNC 1호기 아래 "황삭 공정 → 정삭 공정 → 검사 공정"

    이 구조를 지원하기 위해 두 개의 컬럼을 미리 만들어 놓음:

    상위라인ID: 이 라인이 어떤 상위 라인에 속하는지.
      null이면 독립 라인(최상위), 값이 있으면 하위 작업반.
    계층: "line"(독립 라인) 또는 "sub"(하위 작업반)

    데모에서는 모든 라인이 독립이므로 전부 null/line.

    실무 확장 시 하위 작업반을 추가하려면:
      L03   | CNC 1호기  | 1팀 | null | line | 샤프트 A | 45  | 1.5%
      L03-1 | CNC1 황삭  | 1팀 | L03  | sub  | 샤프트 A | 45  | 1.5%
      L03-2 | CNC1 정삭  | 1팀 | L03  | sub  | 샤프트 A | 45  | 1.0%

    이렇게 하면 line_master만 수정하면 되지만,
    engine.py에 계층 집계 로직(하위 합산 → 상위 판정)을 추가해야 함.
    이 부분은 데모 범위에 포함하지 않으며, 15장(실제 현장 적용)에 명시.

  [라인 추가/삭제 방법]
    line_master.xlsx에 행 추가/삭제 → 코드 수정 없음.
    engine.py가 매 실행마다 line_master를 새로 읽으므로 즉시 반영.


[production_week.xlsx — 주간 raw data]

  이번 주(월~일)의 모든 생산 데이터가 시간별로 쌓이는 파일.
  매시간 N행(라인 수만큼)이 추가되어, 주말까지 최대 약 60N행(데모: 720행).

  컬럼 구조 (25개):

  식별 (7):
    날짜, 시간, 라인ID, 라인명, 팀, 생산품목, 일일목표
  시간당 지표 (5):
    시간당생산, 시간당양품, 시간당불량, 시간당폐기, 시간당불량률(%)
  누적 지표 (5):
    누적생산, 누적양품, 누적불량, 누적폐기, 누적불량률(%)
  진척 지표 (3):
    달성률(%), 예상달성률(%), 달성갭(%p)
  가동 지표 (4):
    시간당가동(분), 시간당비가동(분), 시간당가동률(%), 누적가동률(%)
  기타 (1):
    이상플래그 (시뮬레이터에서 어떤 이상이 심어져 있는지 표시, 운영 시 빈칸)

  [날짜/시간 컬럼을 분리한 이유]
    Power BI에서 날짜별 슬라이서(달력 필터)와 시간대별 X축을 별도로 활용 가능.
    "같은 시간대(10시)의 요일별 비교" 같은 분석도 가능해짐.
    n8n에서 "오늘 날짜의 마지막 2개 시간" 필터링도 간결해짐.

  [주간 파일로 관리하는 이유]
    하루 단위 파일이면 Power BI에서 "최근 3일 추이"를 보려면 파일 3개를 참조해야 함.
    월 단위 파일이면 한 파일에 ~3,600행이 쌓여서 과도함.
    주간 단위(720행)가 Power BI 연결과 파일 크기 사이의 적절한 균형점.

  [이 파일은 n8n이 만들지 않음]
    실제 현장에서는 MES/ERP에서 Excel을 생성하여 SharePoint에 자동 업로드함.
    데모에서는 테스트 시뮬레이터(워크플로 A)가 data_bank에서 행을 꺼내서
    이 파일에 append하는 방식으로 대체.
    매주 월요일 08:00에 새 파일이 외부에서 생성된다고 가정함.


[이상 시나리오 5가지]

  데모 데이터(data_bank.xlsx)에 의도적으로 심어놓은 이상 상황.
  테스트 시 이상 탐지가 실제로 작동하는지 확인하는 용도.

  1. CNC 1호기 설비 고장 (11:00~14:00)
     11시 생산 급감 → 12~13시 생산 0 → 14시 시운전 복구
     탐지 포인트: 생산량 급감 + 가동률 저하 + 2시간 연속 생산 0

  2. 도장 라인 불량률 급등 (09:00~12:00)
     원자재 문제로 불량률 3%→24% → 12시 교체 후 개선
     탐지 포인트: 불량률이 정상의 8배

  3. 사출 1호기 금형 이상 (13:00~15:00)
     생산속도 50% 감소, 서서히 회복
     탐지 포인트: 생산량 급감 + 달성갭 마이너스 전환

  4. 조립 2라인 작업자 교체 (10:00~12:00)
     생산속도 저하 + 불량 증가 동시 발생
     탐지 포인트: 복합 이상 (생산↓ + 불량↑ 동시)

  5. 프레스 1호기 자재 대기 (15:00~17:00)
     가동률 100%→20% 급락, 17시 일부 입고
     탐지 포인트: 가동률 급락 + 비가동시간 급증

5. 저장소 구조
================================================================================

모든 데이터를 SharePoint에 통일.
이유: Power BI Service가 SharePoint Excel을 공식 지원하므로 연결이 안정적이고,
M365 E3에 포함되어 추가 비용이 없음.

[/production/] — 운영 데이터 (Power BI가 읽는 파일)

  production_week.xlsx — 이번 주 raw data
    내용: 4장에서 정의한 25개 컬럼, 시간별 N행 누적
    용도: n8n이 읽고 분석 / Power BI가 라인별 시간대별 상세 차트에 사용
    갱신: 매시간 외부에서 행 추가 (데모: 시뮬레이터가 append)
    크기: 주 최대 약 60N행 (데모: 720행)

  production_results.xlsx — 분석 결과 (3개 탭)

    Tab 1: hourly_summary
      매시간 n8n이 팀별로 집계하여 N행 추가 (팀 수만큼, 데모: 3행)
      컬럼: 날짜, 시간, 팀, 작업건수, 계획합계, 실적합계, 달성률,
            불량합계, 폐기합계, 평균가동률, 비가동합계
      용도: Power BI 페이지 1 (팀별 시간대별 진척 차트)

      [왜 팀별 요약을 따로 저장하는가]
        production_week.xlsx에 라인별 상세가 있지만,
        Power BI에서 "팀 전체 달성률 추이"를 보려면 매번 라인을 합산해야 함.
        미리 팀 단위로 집계해놓으면 Power BI 성능이 좋고 차트 설정도 간단.

    Tab 2: anomaly_log
      이상 감지 시에만 n8n이 행 추가
      컬럼: 날짜, 시간, 라인ID, 라인명, 팀, 품목,
            type, severity, detail,
            pattern_type, recurrence_count, ai_insight, ai_parsed
      - severity: "심각" / "중간" / "낮음"
      - pattern_type: "신규" / "반복" / "악화" (6장에서 상세 설명)
      - ai_insight: Claude AI가 생성한 분석 텍스트 (심각/악화 건만 있음, 나머지 빈칸)
      - ai_parsed: boolean. AI 응답을 JSON으로 정상 파싱했으면 true, 실패하면 false.
        false인 경우 ai_insight에 비구조화 텍스트가 들어가므로
        Power BI에서 필터링하여 별도 확인 가능.
      용도: Power BI 페이지 2 (이상 탐지 이력, AI 분석)

    Tab 3: daily_summary
      하루 N행 (팀 수만큼, 데모: 3행)
      워크플로 C가 주간 백업 시 생성하거나, 워크플로 D가 일일 리포트 시 생성
      컬럼: 날짜, 팀, 총계획, 총실적, 일일달성률,
            총불량, 총폐기, 일일불량률,
            평균가동률, 총비가동시간,
            이상건수_심각, 이상건수_중간, 이상건수_낮음,
            반복이상건수, 악화이상건수
      집계 방식:
        총계획/총실적/총불량/총폐기/총비가동시간: SUM
        일일달성률: 총실적 / 총계획 (재계산. 시간별 달성률의 평균이 아님.
          이유: 08시 달성률 10% + 17시 달성률 95%의 평균 52.5%는 의미 없음.
          실제 일일 달성률은 총실적/총계획 = 95%)
        평균가동률: 총가동시간 / 총시간 (가중평균. 단순평균은 비가동 시간이 긴
          시간대의 영향을 과소평가함)
        이상건수: anomaly_log에서 해당 날짜+팀 기준 COUNT
      용도: Power BI 페이지 3 (장기 트렌드)

    Tab 4: error_log
      데이터 품질 경고 + 시스템 에러를 기록하는 탭.
      컬럼: 날짜, 시간, workflow, step, error_type, error_message, action_taken
      anomaly_log와 분리하는 이유:
        anomaly_log에는 "생산 이상"만 저장해야 Power BI에서
        "이번 주 심각 이상 5건" 통계에 데이터 에러가 섞이지 않음.
      예시:
        2026-03-23 | 11:00 | B:모니터 | 데이터검증 | data_null | L03 시간당생산 빈칸 | 안내 이메일 발송
        2026-03-23 | 11:00 | B:모니터 | Claude API | timeout | 30초 초과 | 폴백 메시지 생성
      용도: 에러 빈도 추적. 반복 에러 패턴 발견 시 시스템 점검 신호.
      Power BI 연결: 선택적

[/config/] — 설정 파일

  line_master.xlsx
    라인/팀 구성 설정. engine.py가 매 실행마다 읽음.
    라인 추가/삭제 시 이 파일만 수정 → 코드 변경 없음.
    Power BI에서도 조인용으로 연결 (라인 마스터 정보 표시)

  rules.json
    이상 탐지 룰 설정. engine.py가 매 실행마다 읽음.
    룰 추가/수정 시 이 파일만 변경 → 코드 변경 없음. (6장에서 상세 설명)

[/backup/] — 주간 백업

  매주 월요일 07:50에 워크플로 C가 production_week.xlsx를 이 폴더로 이동.
  파일명: production_{시작월요일}_{종료일요일}.xlsx
    예: production_20260323_20260329.xlsx
  이유: 파일명에 기간이 들어있어서 "이 파일이 어느 주 데이터인지" 바로 확인 가능.
  보관 기간: 운영 정책에 따라 결정. 데모에서는 무기한 보관.
  Power BI 연결: 없음. 필요 시 수동으로 열어서 분석.

[/simulator/] — 테스트 전용

  data_bank.xlsx
    데모용 사전 데이터 (N라인 x 10시간 행, 데모: 120행)
    이상 시나리오 5가지가 심어져 있음
    워크플로 A(테스트 시뮬레이터)가 여기서 시간별 데이터를 꺼내서
    production_week.xlsx에 append함
    실무에서는 사용하지 않음 (MES/ERP에서 직접 데이터가 들어오므로)

6. 이상 탐지 방법론

[탐지 구조: Config-driven Rules]

  이상 탐지 룰을 Python 코드에 직접 쓰지 않고,
  JSON 설정 파일(rules.json)과 범용 엔진(engine.py)으로 분리함.

  이유: 나중에 "불량률 기준을 3배에서 2배로 바꾸고 싶다"거나
  "새로운 탐지 조건을 추가하고 싶다"고 할 때 코드를 고치지 않고
  rules.json만 수정하면 즉시 반영됨.
  engine.py가 매 실행마다 rules.json을 새로 읽기 때문.

  rules.json: 어떤 컬럼을, 어떤 방식으로, 어떤 기준값과 비교할지 정의
  engine.py: rules.json을 읽어서 자동 실행하는 범용 코드

  engine.py 동작:
    1) line_master.xlsx 읽기 → 라인 목록, 팀 목록, 라인별 기준값 동적 파악
    2) rules.json 읽기 → 룰 목록 로드
    3) production_week.xlsx에서 마지막 2개 시간대 추출
       (같은 파일 안에 모든 시간이 쌓여 있으므로 별도 previous 탭 불필요)
    4) 각 라인 x 각 룰 조합으로 이상 판정
    5) 결과를 anomaly_log에 저장


[심각도 3단계: 하나의 룰에 3개 레벨]

  하나의 이상 유형(예: 생산량 급감)에 대해
  "얼마나 심각한지"를 수치 범위로 3단계 구분함.

  이유: 생산량이 50% 줄어든 것과 10% 줄어든 것은 대응 수준이 다름.
  50% 감소 → 설비 고장 가능성, 즉시 점검 필요 (심각)
  25% 감소 → 주의 필요, 추이 모니터링 (중간)
  8% 감소 → 참고 사항, 정상 변동 범위 경계 (낮음)

  rules.json 형식:
  {
    "id": "rule_01",
    "name": "생산량 급감",
    "type": "drop_rate",
    "target": "hourly_production",
    "compare_with": "previous",
    "levels": [
      {"severity": "심각", "min": 40, "max": null},
      {"severity": "중간", "min": 20, "max": 40},
      {"severity": "낮음", "min": 5,  "max": 20}
    ],
    "message": "{line_name}: 생산량 {change}% 감소"
  }

  판정 예시:
    변화량 50% → min 40 이상 → "심각"
    변화량 30% → 20~40 사이 → "중간"
    변화량 10% → 5~20 사이 → "낮음"
    변화량 3% → 5 미만 → 정상 (이상 아님, 기록하지 않음)

  신규 조건 추가: rules.json에 블록 추가 → engine.py 수정 없음
  기존 조건 수정: rules.json 숫자 변경 → engine.py 수정 없음
  심각도 기준 변경: levels 배열의 min/max 변경 → engine.py 수정 없음


[탐지 기준 8가지]

  1. 생산량 급감 (drop_rate)
     심각: 40%+ 감소 / 중간: 20~40% / 낮음: 5~20%

  2. 가동률 저하 (below_threshold)
     심각: 30% 미만 / 중간: 30~50% / 낮음: 50~70%

  3. 불량률 급등 (exceeds_baseline)
     심각: 정상의 5배+ / 중간: 3~5배 / 낮음: 2~3배
     ※ "정상불량률"은 line_master.xlsx에서 라인별로 읽음

  4. 달성률 지연 (below_threshold)
     심각: 갭 -30%p 이하 / 중간: -15~-30%p / 낮음: -5~-15%p

  5. 생산 정체 (consecutive_zero)
     심각: 3시간+ 연속 0 / 중간: 2시간 / 낮음: 1시간
     ※ 주간 파일에 모든 시간 데이터가 있으므로
       같은 파일에서 해당 라인의 최근 N시간을 조회하면 판정 가능

  6. 폐기 발생 (above_threshold)
     심각: 폐기율 3%+ / 중간: 1~3% / 낮음: 1% 미만 발생

  7. 복합 이상 — 생산 감소 + 불량 상승 동시 (compound)
     심각: 30%↓ + 3배↑ / 중간: 15%↓ + 2배↑ / 낮음: 5%↓ + 1.5배↑

  8. 가동률 급락 (drop_points)
     심각: 50%p+ 하락 / 중간: 30~50%p / 낮음: 15~30%p


[compound(복합) 룰 JSON 형식]

  두 가지 조건이 동시에 충족될 때만 이상으로 판정.
  예: "생산량도 줄고 불량률도 올랐다" = 작업자 교체, 설비 이상 등 복합 원인 의심

  {
    "id": "rule_07",
    "type": "compound",
    "operator": "AND",
    "levels": [
      {
        "severity": "심각",
        "conditions": [
          {"type": "drop_rate", "target": "hourly_production", "threshold": 30},
          {"type": "exceeds_baseline", "target": "hourly_defect_rate", "multiplier": 3}
        ]
      },
      {
        "severity": "중간",
        "conditions": [
          {"type": "drop_rate", "target": "hourly_production", "threshold": 15},
          {"type": "exceeds_baseline", "target": "hourly_defect_rate", "multiplier": 2}
        ]
      }
    ]
  }

  현재 지원 범위: AND 조합, 2개 조건, 1단계(flat) 구조만.
  OR 조합, 3개 이상 중첩, compound 안의 compound(재귀)는 미지원.
  이유: 복잡한 조합은 실무에서도 드물고, 구현·테스트 비용 대비 효과가 낮음.
  실무 확장 시 필요하면 engine.py에 OR 로직과 재귀 파싱을 추가할 수 있음.


[비교 타입 7가지]

  drop_rate: 이전 대비 N% 감소 (예: 생산량 40% 급감)
  exceeds_baseline: 기준값의 N배 초과 (예: 불량률이 정상의 3배)
  below_threshold: 고정값 미만 (예: 가동률 50% 미만)
  above_threshold: 고정값 초과 (예: 폐기율 3% 초과)
  consecutive_zero: 연속 N회 0 (예: 2시간 연속 생산 0)
  drop_points: 이전 대비 N%p 하락 (예: 가동률 30%p 급락)
  compound: 두 조건 동시 충족

  이 7가지는 제조 생산 데이터에서 필요한 비교 방식을 커버함.
  다른 도메인(물류, 에너지, 금융 등)에서는 이동평균 비교, 시간대별 기대값 비교 등
  추가 비교 타입이 필요할 수 있으며, 이 경우 engine.py에 새 type을 추가해야 함.


[알림 분기]

  모든 이상 → anomaly_log에 저장 (심각/중간/낮음 전부 기록)

  낮음:
    - 이메일 없음, 로그만 저장
    - 다음날 08:00 일일 리포트에 포함됨
    - 단, "악화"(7일 내 3회+)로 분류되면 심각으로 에스컬레이션
      이유: 사소한 이상도 반복되면 근본 원인이 있을 수 있음

  중간:
    - 이메일 1회 발송 (같은 사이클의 낮음도 "참고" 섹션에 포함)
    - "악화"이면 심각으로 에스컬레이션
    - 이유: 관리자가 인지는 해야 하지만 즉각 대응은 불필요

  심각:
    - AI 호출 (Claude API) + 이메일 발송
    - 이메일에 같은 사이클의 중간+낮음도 포함
    - 미해결 시 30분 후 에스컬레이션 반복 (심각만)
    - 이유: 즉각 대응이 필요하고, 안 고쳐졌으면 다시 알려야 함

  이메일 구조:
    ━━━ [심각] ━━━
    CNC 1호기 — 생산량 91.7% 급감 (AI 분석: ...)
    ━━━ [중간] ━━━
    도장 라인 — 불량률 9.9%
    ━━━ 참고 (낮음) ━━━
    - 프레스 2호기: 생산량 8% 감소
    - 사출 2호기: 달성률 -3.2%p

  이유: 심각 이메일을 받았을 때 "다른 곳도 좀 이상하다"는 맥락을
  낮음 항목으로 함께 보여주면 관리자가 전체 상황을 파악하기 쉬움.


[이상 패턴 분류: Python + AI 하이브리드]

  같은 문제가 반복되는지 확인하여 "처음인지, 또 발생한 건지, 점점 심해지는지" 분류.
  이유: "오늘 처음 발생한 문제"와 "이번 주에 3번째 발생한 문제"는 대응이 다름.
  반복되면 단순 수리가 아니라 근본 원인(설비 교체, 공정 변경 등)을 봐야 함.

  1단계 — Python (모든 이상에 적용, AI 호출 없음, 비용 0원):
    anomaly_log에서 최근 7일 이력 조회
    동일 라인+유형이 몇 번 있었는지 COUNT
    "신규": 0회 / "반복": 1~2회 / "악화": 3회+ 또는 심각도 상승 추세

    이유: 숫자 세기이므로 AI가 필요 없음. 모든 이상에 적용해도 비용 증가 없음.

  2단계 — Claude API (심각 또는 악화 건만):
    Python 1차 분류 결과 + 이력을 마스킹하여 전달
    AI가 1차 분류를 검증("동의" 또는 "보정 + 이유")
    근본 원인 제언 생성 (반복/악화 시)

    이유: 모든 건에 AI를 부르면 비용이 3~5배 증가.
    중요한 건만 AI가 검증하면 비용을 최소화하면서 AI의 가치를 유지.


[AI 프롬프트: 동시 다발 이상 일괄 처리]

  한 사이클에 심각+악화 건이 여러 개이면 1번의 API 호출로 묶어서 처리.

  이유: 개별 호출(3건이면 3회)보다 비용이 1/3.
  또한 "CNC 고장 때문에 다음 공정도 정체"같은 이상 간 연관 분석은
  한 프롬프트에 여러 건이 있어야 AI가 연결할 수 있음.

  출력은 JSON 형식으로 요청:
  {
    "anomalies": [
      {"id": 1, "summary": "...", "root_cause": "...", "action": "...",
       "pattern_verification": "동의/보정", "cross_impact": "..."}
    ],
    "overall_assessment": "전체 상황 종합"
  }

  cross_impact: "LINE_A 고장이 LINE_B 정체 원인일 수 있음" 같은 연관 분석

  [AI 출력 파싱]
    1차: JSON 파싱 시도 (Claude가 ```json으로 감쌀 수 있으므로 래퍼 제거 후 파싱)
    2차: 파싱 실패 시 전체 텍스트를 ai_insight에 저장 (폴백)
    ai_parsed 컬럼으로 성공/실패 기록

    이유: Claude가 항상 완벽한 JSON을 돌려주는 건 아님.
    폴백이 있으면 파싱 실패해도 시스템이 죽지 않고 텍스트라도 저장됨.

7. 보안 전략
================================================================================

"회사 데이터를 AI에 보내도 되는가?"에 대한 답.
4단계 보안 레벨을 제공하여 보안 요구 수준에 따라 선택 가능.

[레벨 1: 기본] 집계 데이터만 전달
  숫자만 보냄 ("생산량 48→4, 가동률 15%"). 회사명, 라인명 미포함.

[레벨 1+: 권장] 마스킹 처리
  실제 이름을 코드로 치환: "CNC 1호기"→"LINE_A", "샤프트 A"→"PROD_01"
  AI 응답 후 코드를 원래 이름으로 복원.
  AI는 "LINE_A"가 뭔지 알 수 없음.
  치환 사전(mask_dict)은 n8n Code 노드 안에 보관, 외부 전송 안 됨.

  [상세 설명]
  mask_dict 생성: line_master에서 동적으로 생성.
  라인 추가 시 mask_dict도 자동 확장. 하드코딩 아님.
  예: {"CNC 1호기": "LINE_A", "샤프트 A": "PROD_01", ...}

  언마스킹: AI 응답 텍스트에서 단순 문자열 치환 (string replace).
    예: 응답의 "LINE_A" → "CNC 1호기"로 복원.

  AI가 마스킹 코드를 사용하지 않는 경우 대비:
    프롬프트에 "응답에서 반드시 제공된 코드(LINE_A 등)를 사용하세요" 명시.
    언마스킹되지 않은 코드는 그대로 남음 (깨지지 않고 코드명이 표시됨).

  실무 적용 시: mask_dict를 환경변수 또는 별도 보안 저장소(Key Vault)로 분리 권장.
  현재 n8n Code 노드에 평문으로 있으면 n8n UI 접근자가 볼 수 있으므로.

[레벨 2: 옵션] LLM 없이 템플릿 모드
  AI를 아예 안 쓰고, 조건에 따라 미리 만든 문구를 자동 선택.
  외부 데이터 전송 0건.
  단점: 근본 원인 제언, 교차 분석 같은 맥락적 해석 불가.

[레벨 3: 확장] 로컬 LLM (Ollama)
  사내 서버에서 LLM 직접 실행. 데이터가 회사 밖으로 나가지 않음.
  단점: 서버 필요, 응답 품질이 Claude/GPT 대비 낮을 수 있음.

[기능별 보안 정리]
  Python 패턴 분류      → 외부 전송 없음. n8n 내부에서 로컬 실행.
  Claude API 해석       → 마스킹 데이터만 전송. 학습 미사용 약관 적용.
  Power BI AI 시각화    → 외부 전송 없음. Microsoft 테넌트 내 ML.NET 처리.
                         E3 약관: "고객 데이터를 AI 학습에 사용하지 않음"

[워크플로 D: 일일 리포트] — Cron 매일 08:00 — ★ AI Agent 방식

  목적: 전날 발생한 모든 이상(심각+중간+낮음)을 AI Agent가 자율적으로 분석하여
  상황에 맞는 깊이의 리포트를 생성하고 이메일 발송.
  이유: 낮음 심각도는 실시간 이메일이 안 가므로 일일 리포트에서 확인.
  또한 하루 전체를 종합적으로 보는 리포트가 관리자에게 필요.

  [기존 방식과의 차이 — 왜 AI Agent인가]

    기존(단발 호출): 코드가 데이터를 미리 집계 → Claude에 "요약해줘" 1회 호출 → 끝
    Agent 방식: Claude에게 도구(Tool)를 주고 목표만 전달 → AI가 스스로 판단하여
                필요한 데이터를 조회하고, 결과를 보고 추가 조회 여부를 결정하고,
                상황의 심각성에 따라 리포트 깊이를 자율 조절함.

    핵심 차이:
      - 기존: 개발자가 코드로 분석 순서를 고정 → 항상 같은 포맷의 리포트
      - Agent: AI가 스스로 다음 행동을 결정 → 조용한 날은 간단히, 심각한 날은 상세하게

    Agent를 여기에 적용한 이유:
      1. 워크플로 D는 핵심(B)과 독립 → 실패해도 실시간 알림에 영향 없음
      2. 리포트 작성은 "상황을 보고 판단하는" 일이라 Agent 패턴에 자연스러움
      3. 하루 1회 실행이라 비용 증가 최소 (월 +$0.2~0.5)

  [AI Agent 동작 원리]

    Agent는 Claude API의 Tool Use 기능을 사용하여 구현.
    Agent에게 도구(데이터 조회 함수)와 목표(리포트 작성)를 주면,
    AI가 스스로 어떤 도구를 어떤 순서로 쓸지 결정하고,
    결과를 보고 추가 조회가 필요한지 판단하는 루프를 반복함.

    while True:
      response = claude_api.call(messages, tools)
      if response.stop_reason == "tool_use":
        # AI가 "이 도구를 써야 해"라고 요청 → 실행 후 결과 전달
        tool_result = execute_tool(response.tool_call)
        messages.append(tool_result)
        continue  # AI가 다시 생각
      elif response.stop_reason == "end_turn":
        # AI가 "다 했다"고 판단 → 최종 리포트 출력
        final_report = response.text
        break

  [Agent에게 제공하는 도구 4가지]

    도구 1: get_anomaly_log(start_date, end_date, severity?, line_id?)
      anomaly_log에서 조건에 맞는 이상 이력 조회.
      Agent가 "어제 심각만 보자" 또는 "CNC 라인의 7일 이력을 보자"고 판단 시 사용.

    도구 2: get_daily_summary(start_date, end_date, team?)
      daily_summary에서 일별 요약 조회.
      Agent가 "지난주와 비교해야겠다"고 판단 시 사용.

    도구 3: get_line_master()
      라인/팀 구성 정보 조회.
      Agent가 "이 라인이 어느 팀인지, 정상 불량률이 얼마인지" 확인 시 사용.

    도구 4: get_hourly_detail(date, line_id?)
      production_week에서 특정 날짜의 시간별 상세 데이터 조회.
      Agent가 "CNC의 어제 시간대별 변화를 보고 싶다"고 판단 시 사용.

    각 도구는 Python 함수로 구현하며, SharePoint에서 데이터를 읽어 반환함.
    모든 도구의 반환값은 마스킹 적용 후 Agent에게 전달됨.

  [Agent의 자율 판단 예시]

    사례 1 — 심각한 날:
      Agent: get_anomaly_log(어제) → 심각 3건 확인
      Agent: "지난주보다 많은지 비교해야지" → get_anomaly_log(지난주, 심각)
      Agent: "CNC에서 2건, 이력 확인" → get_anomaly_log(7일, line_id=L03)
      Agent: "점점 악화되고 있다. 상세 리포트 + 경영진 CC 추천"
      → 도구 4회 사용, 상세 리포트 출력

    사례 2 — 조용한 날:
      Agent: get_anomaly_log(어제) → 낮음 2건만
      Agent: "특이사항 없다. 간단히 정리하면 충분하다"
      → 도구 1회 사용, 1~2줄 요약 출력 (비용 절약)

  [리포트 깊이 기준 — System Prompt에 명시]

    간단: 심각 0건 + 중간 2건 이하 → 요약 1~2줄
    보통: 심각 1~2건 또는 중간 3건 이상 → 상세 분석 포함
    상세: 심각 3건 이상 또는 악화 추세 → 추세 분석 + 경영진 CC 추천

  [Agent 출력 형식]

    {
      "report_level": "간단/보통/상세",
      "cc_management": true/false,
      "summary": "한 줄 요약",
      "sections": {
        "핵심_경고": "...",
        "추세_분석": "...",
        "기타_이상": "...",
        "권장_조치": "..."
      },
      "recommendations": ["권장 조치 1", "권장 조치 2"]
    }

  [안전장치]

    1. 도구 호출 최대 8회 — 무한 루프 방지. 초과 시 지금까지 데이터로 강제 생성.
    2. API 실패 시 폴백 — 기존 고정 포맷 리포트로 대체 발송.
    3. 파싱 실패 시 — Agent 텍스트 출력을 그대로 이메일에 포함.
    4. 도구 실행 실패 — Agent에게 "조회 실패" 메시지 전달, 가능한 범위로 작성.
    5. API 타임아웃 — 60초 제한.

  [비용 영향]

    기존(단발 호출): 하루 1회 × ~2,000 토큰 ≈ 월 ~$0.1
    Agent 방식: 하루 2~6회 × ~5,000~15,000 토큰 ≈ 월 ~$0.3~0.8
    비용 증가: 월 +$0.2~0.5 (조용한 날 2회, 바쁜 날 5~6회 → AI가 자율 조절)

  흐름:
    Cron Trigger (매일 08:00)
      → Python: 도구 함수 정의 (get_anomaly_log, get_daily_summary 등)
      → Python: 마스킹 사전(mask_dict) 준비
      → Anthropic: AI Agent 루프 시작 (System Prompt + tools)
        → Agent가 도구 호출 → Python이 실행 → 결과(마스킹됨) 반환
        → Agent가 결과 분석 → 추가 도구 호출 또는 최종 리포트 출력
        → (반복, 최대 8회)
      → Python: Agent 출력 JSON 파싱 (실패 시 텍스트 폴백)
      → Python: 복원(언마스킹) + 리포트 HTML 생성
      → Gmail: 일일 리포트 이메일
        (report_level="상세" + cc_management=true이면 경영진 CC)

  리포트 내용 (Agent가 상황에 따라 자율 구성):
    전날 총 이상 건수 (심각 N건, 중간 N건, 낮음 N건)
    심각 상세 (각 건의 AI 분석 포함)
    중간 요약
    낮음 리스트
    반복/악화 하이라이트
    추세 분석 (심각 건이 있을 때 Agent가 자율적으로 추가)
    AI 종합 코멘트 (개선 제언)
    Power BI 대시보드 링크

────────────────────────────────────────────────────────────

[중복 알림 방지]

  저장: n8n Static Data (워크플로 내부 JSON 메모장)
  형식: {"L03_생산량급감": "2026-03-23T11:00:00", ...}
  키: 라인ID + 이상유형 조합 (라인마다, 유형마다 별도 타이머)

  심각:
    → 같은 키가 30분 이내에 있으면 스킵
    → 30분 경과 + 다음 사이클에서 다시 감지(미해결) → 에스컬레이션 알림
    이유: 안 고쳐졌으면 다시 알려야 하지만, 너무 자주 오면 무시하게 됨. 30분이 균형점.

  중간: 사이클당 1회 (30분 반복 없음)
  낮음: 이메일 없음

  서로 다른 이상은 별도로 관리됨.
    예: CNC 생산급감과 도장 불량급등은 다른 키이므로 각각 따로 알림 가능.


================================================================================
9. 에러 핸들링
================================================================================

파이프라인에서 에러가 발생할 수 있는 지점은 7곳이며,
각각에 기초 수준의 대응을 적용함.

[3가지 원칙]
  1. 하나의 단계가 실패해도 전체 워크플로가 멈추지 않을 것
  2. 데이터 보존을 알림보다 우선할 것 (먼저 저장, 그다음 이메일)
  3. 모든 에러를 error_log 탭에 기록하여 사후 확인 가능할 것

[error_log 탭] — production_results.xlsx에 추가 (Tab 4)
  컬럼: 날짜, 시간, workflow, step, error_type, error_message, action_taken
  용도: 에러 빈도 추적, 반복 에러 패턴 발견
  Power BI 연결: 선택적 (에러 모니터링 필요 시)


[에러 지점 1: SharePoint 파일 읽기 실패]

  발생 상황: 파일 미존재, 인증 토큰 만료, 네트워크 불안정
  대응:
    n8n 노드 설정 "On Error" → "Continue Using Error Output"
    Error Output 경로 → error_log에 기록 → 워크플로 종료
  이유: 파일을 못 읽으면 이후 모든 단계가 의미 없음.
    에러를 기록하고 깨끗하게 종료. 60분 뒤 다음 사이클에서 자동 재시도됨.


[에러 지점 2: 데이터 검증 — null/빈칸/라인 누락]

  SAP에서 생산을 계획하고 해당 계획에 따라 생성한 데이터이므로,
  값이 비어있거나 라인이 누락된 것은 "생산 이상"이 아니라 "데이터 전달 문제"임.
  값이 0(숫자)인 것은 실제로 생산을 못 한 것이므로 정상적인 이상 탐지 대상.

  [검증 1: 필드가 null/빈칸인 라인]
    라인 행은 있지만, 시간당생산/시간당불량/시간당가동률 등 핵심 필드가 빈 경우.
    예: L03 행은 있는데 시간당생산 칸이 비어있음.
    → MES가 값을 채우지 못한 데이터 품질 문제.

    대응:
      해당 라인은 이상 탐지에서 제외 (잘못된 데이터로 분석하면 오탐 발생)
      사용자에게 "데이터 누락" 안내 이메일 발송
      error_log에 기록
      나머지 정상 라인은 이상 탐지 정상 진행

  [검증 2: 라인 행 자체가 없는 경우]
    line_master에 등록된 라인인데 이번 시간 데이터에 행이 아예 없는 경우.
    이건 3가지 상황이 있으며, 각각 다르게 처리해야 함:

    상황 1 — 목표 완료:
      해당 라인의 오늘 누적생산이 일일목표 이상이면, 더 만들 필요가 없으므로 정상.
      예: 일일목표 100개, 08시 50개 + 09시 50개 = 누적 100개. 10시에 행 없음 = 정상.
      대응: 아무것도 안 함. 경고 안 보냄.

    상황 2 — MES 데이터 전달 오류:
      누적생산이 일일목표 미만인데 데이터가 끊긴 경우.
      예: 일일목표 450개, 누적 93개인데 10시 데이터 없음 = 아직 357개 남았는데 데이터가 안 옴.
      대응: 사용자에게 "데이터 중단" 안내 이메일.
        내용: "L03 누적 93개 / 목표 450개 (잔여 357개). 마지막 데이터: 09시.
               MES 데이터 전달 상태를 확인해주세요."
      error_log에 기록.

    상황 3 — 라인 미가동:
      오늘 해당 라인의 데이터가 한 건도 없는 경우.
      MES는 생산 활동이 있을 때 데이터를 생성하므로,
      한 번도 데이터가 온 적 없다면 아직 가동을 시작하지 않은 것으로 판단.
      대응: 정상. 안내 안 보냄.
      이유: MES에서 데이터가 올라온다는 가정하에,
        데이터의 존재 여부 자체가 가동 여부를 나타냄.
        불필요한 안내를 보내면 관리자가 알림을 무시하게 됨.

      판정 로직:
        행 없음 + 오늘 이전 데이터 있음 + 누적생산 >= 일일목표  → 상황 1 (목표 완료, 정상)
        행 없음 + 오늘 이전 데이터 있음 + 누적생산 < 일일목표   → 상황 2 (MES 오류, 안내)
        행 없음 + 오늘 데이터가 한 건도 없음                   → 상황 3 (미가동, 정상)

  [검증 3: 필수 컬럼 누락 / 타입 오류]
    Excel 파일에 기대한 컬럼이 없거나 컬럼명이 바뀐 경우.
    예: "시간당생산"이 "시간생산"으로 바뀌었거나, 숫자 컬럼에 텍스트가 들어있음.
    대응: error_log에 기록 → 워크플로 종료.
    이유: 컬럼 구조가 달라지면 모든 분석이 의미 없음. 데이터 소스를 먼저 수정해야 함.

  [값 = 0과 null의 구분]
    값 = 0 (숫자):
      "실제로 한 개도 못 만든 것". 설비 고장, 자재 부족 등 실제 생산 문제.
      이상 탐지 대상으로 정상 진행.
      예: 시간당생산 = 0 → "생산량 급감" 룰에서 이전 대비 100% 감소 → 심각

    값 = null/빈칸:
      "데이터가 안 채워진 것". MES가 값을 전달하지 못한 시스템 문제.
      이상 탐지에서 제외. 사용자 안내만 발송.
      이유: null을 0으로 오해하면 "설비 고장"이라는 오탐이 발생하고,
        null로 나눗셈하면 연산 에러가 남.

  [데이터 검증 결과의 이메일 포함 방식]
    데이터 경고가 있으면, 이상 탐지 이메일의 맨 위에 표시.
    이유: "이 라인은 분석 자체가 안 됐다"는 건 이상보다 더 우선적으로 알아야 할 정보.

    이메일 구조:
      ━━━ [데이터 경고] ━━━
      ⚠ CNC 1호기 (L03): 시간당생산, 시간당가동률 데이터가 비어있습니다
      ⚠ 조립 1라인 (L10): 누적 93개 / 목표 450개인데 이번 시간 데이터 미수신
      → 위 라인은 이상 탐지가 수행되지 않았습니다.

      ━━━ [심각] ━━━
      ... (정상 라인들의 이상 탐지 결과)

    데이터 경고만 있고 이상이 없는 경우에도 이메일 발송.
    이유: 데이터가 안 오는 건 그 자체로 관리자가 알아야 할 상황.

    이상도 없고 데이터 경고도 없는 경우에만 이메일 안 보냄.

  [데이터 경고의 저장 위치]
    anomaly_log가 아닌 error_log에 저장.
    이유: anomaly_log는 "생산 이상"만 담아야 Power BI에서
      "이번 주 심각 이상 5건" 통계에 데이터 에러가 섞이지 않음.


[에러 지점 3: 이상 탐지 중 연산 에러]

  발생 상황: drop_rate 계산 시 이전 값이 0이면 나누기 0,
    line_master에 정상불량률이 빠져있으면 exceeds_baseline 계산 불가.

  대응: 각 룰을 try-except로 감싸서 해당 룰만 스킵하고 나머지 계속 진행.
    이전 값이 0인 경우: 감소율 계산 불가이므로 해당 룰 스킵 (에러 아닌 정상 처리).
      현재 값도 0이면 "변화 없음", 현재 값 > 0이면 "증가" → 둘 다 이상 아님.
    line_master 기준값 누락: 해당 룰 스킵 + error_log 기록.
    예상 못한 에러: 해당 룰 스킵 + error_log 기록.

  핵심 원칙: 하나의 룰 에러가 전체를 멈추면 안 됨.
    rule_01에서 에러가 나도 rule_02~08은 정상 실행되어야 함.

  ※ 값 = 0인 경우는 에러가 아님:
    Phase 2(데이터 검증)을 통과한 데이터는 null이 아닌 숫자만 남아있음.
    값이 0이면 이상 탐지 룰이 정상적으로 "생산량 급감" 등으로 판정함.
    나누기 0 문제는 "이전 시간 값이 0일 때"만 발생하며, 이때는 변화율 계산을
    스킵하는 것이 올바른 처리 (0에서 0이면 변화 없음, 0에서 양수면 증가).


[에러 지점 4: SharePoint 쓰기 실패]

  발생 상황: hourly_summary, anomaly_log 저장 시 파일 잠금,
    권한 문제, SharePoint 용량 초과.

  대응:
    n8n 노드 설정:
      Retry on Fail: ON
      Max Retries: 2
      Wait Between Retries: 10초
    2회 재시도 후에도 실패 → error_log 기록 → 이메일은 여전히 발송 시도.

  이유: SharePoint 쓰기 실패는 일시적인 경우가 많음 (파일 잠금, 순간 끊김).
    10초 후 재시도하면 대부분 해결됨.
    쓰기는 실패해도 이메일은 보내야 함.
    "anomaly_log에 못 적었지만 관리자에게는 알려야" 하기 때문.


[에러 지점 5: Claude API 호출 실패]

  발생 상황: API 키 만료, 요금 부족, 서버 장애, 네트워크 타임아웃, 속도 제한.

  대응:
    n8n Anthropic 노드 설정:
      Timeout: 30초
      Retry on Fail: ON
      Max Retries: 1
      Wait Between Retries: 15초
    재시도 후에도 실패 → AI 없이 폴백 메시지 생성.

  폴백 메시지:
    AI 분석 텍스트 대신 템플릿 기반 메시지를 자동 생성.
    예: "[AI 분석 불가 - 자동 생성 메시지]
         CNC 1호기: 생산량 급감 감지. 시간당생산 48→4 (91.7% 감소).
         최근 7일 내 3회 반복 발생."
    이메일 제목에 "[AI 분석 불가]" 표시하여 수신자가 구분 가능.

  이유: AI가 안 되더라도 이상 감지 자체는 Python이 이미 완료한 상태.
    "AI 분석은 못했지만 문제는 있으니 알려드립니다"가 "아무것도 안 보내는 것"보다 나음.


[에러 지점 6: AI 응답 파싱 실패]

  발생 상황: Claude가 JSON을 돌려줬는데 형식이 틀리거나,
    요청한 필드가 빠져있거나, JSON이 아닌 텍스트를 돌려줌.

  대응:
    1차: JSON 파싱 시도 (```json 래퍼 제거 후)
    2차: 필수 필드 검증 (anomalies 배열 존재, 건수 일치 등)
    3차: 파싱 실패 시 전체 텍스트를 ai_insight에 그대로 저장 (폴백)
    ai_parsed 컬럼으로 성공(true)/실패(false) 기록.

  이유: AI 응답이 완벽한 JSON이 아니어도 텍스트 자체는 유용한 정보.
    파싱 실패해도 시스템이 죽지 않고 텍스트라도 저장해놓으면
    Power BI에서 읽을 수 있음. ai_parsed=false 필터로 실패 건만 확인 가능.


[에러 지점 7: Gmail 발송 실패]

  발생 상황: Gmail 인증 만료, 수신자 주소 오류, 일일 발송 한도 초과.

  대응:
    n8n Gmail 노드 설정:
      Retry on Fail: ON
      Max Retries: 2
      Wait Between Retries: 10초
    재시도 후에도 실패 → error_log에 기록.
    이상 탐지 결과는 이미 anomaly_log에 저장된 상태이므로 데이터 손실 없음.

  이유: 이메일 실패가 전체 워크플로를 멈추면 안 됨.
    anomaly_log + hourly_summary 저장까지는 완료되었으므로
    이메일만 실패한 건 "알림 채널 장애"이지 "분석 실패"가 아님.


[에러 핸들링 전체 흐름 요약]

  워크플로 B 실행 순서에서 각 에러 대응:

  파일 읽기 실패      → error_log 기록 → 종료 (이후 단계 실행 안 함)
  데이터 검증 실패     → 문제 라인 분리 → 안내 이메일 + 정상 라인만 계속 진행
  이상 탐지 연산 에러  → 해당 룰만 스킵 → 나머지 룰 계속 진행
  SharePoint 쓰기     → 2회 재시도 → 실패해도 이메일은 발송
  Claude API 실패     → 1회 재시도 → 실패 시 폴백 메시지로 이메일 발송
  AI 파싱 실패        → 전체 텍스트 저장 (폴백)
  Gmail 발송 실패     → 2회 재시도 → 실패해도 데이터는 이미 저장됨

10. Power BI 대시보드

[연동 방식]
  데이터: SharePoint /production/ 내 Excel 2개 + /config/line_master.xlsx
  자동 새로고침: 하루 8회 (Power BI Pro, E3 포함)
  이유: 긴급 알림은 이메일로 60분 이내 도착하므로 Power BI 새로고침 주기와 무관.
  Power BI는 "더 자세히 보고 싶을 때" 여는 상세 분석 도구.

[페이지 1: 실시간 현황]
  데이터: hourly_summary + production_week.xlsx
  - 팀별 달성률, 가동률 KPI 카드
  - 라인별 시간대별 생산량/가동률 꺾은선 차트
    (production_week.xlsx에 라인별 상세가 있으므로 가능)
  - Key Influencers: "달성률이 낮아지는 데 가장 큰 영향을 주는 요인" 자동 분석
    ※ Power BI 내장 AI 기능. 최소 300행 축적 후 안정적 결과 (약 3일치).
  - Smart Narrative: 같은 페이지 차트를 참조한 AI 텍스트 요약

[페이지 2: 이상 탐지 & AI 인사이트]
  데이터: anomaly_log
  - 심각도별 KPI 카드 + 라인별 분포 차트
  - 이상 이력 타임라인 + pattern_type(신규/반복/악화) 표시
  - 필터: "반복만" / "심각만" / 특정 라인
  - Decomposition Tree: 이상건수를 severity → type → 라인 → pattern으로 드릴다운
    이유: "심각 이상이 어떤 유형이고 어떤 라인에서 많이 나는지" 한눈에 파악
  - 선택 건의 AI 해석 텍스트 표시 (드릴스루 또는 툴팁)

[페이지 3: 장기 트렌드]
  데이터: daily_summary
  - 일별/주별 달성률, 불량률, 이상 빈도 추이
  - Smart Narrative: 주간/월간 추이 AI 텍스트 요약
  - 이유: "이번 주가 지난 주보다 나아졌는지" 같은 추세를 확인하는 경영진용

[Power BI AI 시각화 보안]
  Key Influencers, Decomposition Tree, Smart Narrative:
  Power BI 내장 기능. Microsoft ML.NET 기반 통계 알고리즘으로
  Power BI Service 테넌트 내에서 처리. 외부 AI 서비스에 데이터를 전송하지 않음.
  E3 엔터프라이즈 약관: "고객 데이터를 AI 학습에 사용하지 않음"

11. 데이터 수명주기
================================================================================

주간 파일 단위로 회전하는 단순한 구조.
이유: 행 단위로 "이 행은 30일 됐으니 압축, 이건 180일 됐으니 삭제"
하는 방식은 로직이 복잡하고 에러 가능성이 높음.
파일 통째로 이동하는 방식이 단순하고 안전.

[현재 주]
  production_week.xlsx: 라인별 시간대별 상세 (최대 ~720행)
  Power BI 직접 연결 → 실시간 현황 차트

[지난 주~]
  /backup/ 폴더에 파일 단위 보관
  Power BI 연결 안 함 (필요 시 수동으로 열어서 분석)

[장기 트렌드]
  daily_summary 탭에 일별 요약이 계속 누적 (월 ~90행)
  Power BI 페이지 3에서 사용
  이유: 주간 raw data를 장기간 볼 필요는 거의 없지만,
  "월별 달성률 추이"는 경영진이 필요로 하므로 요약만 보관

[보관 정책]
  /backup/ 파일: 운영 정책에 따라 결정. 데모에서는 무기한 보관.
  daily_summary: 무기한 누적 (행 수가 작으므로 문제없음)
  실무 적용 시: /backup/ 파일은 3~6개월 후 삭제 또는 별도 아카이브 서버로 이동

13. n8n 워크플로 자동 생성 도구
================================================================================

n8n 워크플로를 GUI에서 수동으로 만들면 노드 연결/설정 실수가 발생하기 쉬움.
미리 만들어둔 JSON 템플릿을 n8n에 Import하면 한 번에 세팅 가능.

[Phase 1: Claude Code Skill] ★ MVP

  도구: /n8n-gen
  방식: references/ 디렉토리에 완성된 워크플로 JSON 템플릿 보관
  Credential(인증 정보)은 n8n 인스턴스마다 다르므로 placeholder로 처리:
    "credentials": {"sharepointExcel": {"id": "PLACEHOLDER", "name": "REPLACE_ME"}}

  사용법:
    1. Skill 호출 → JSON 파일 생성
    2. n8n UI에서 Import (Settings → Import from File)
    3. 각 노드의 Credential 아이콘 클릭 → 본인의 Credential 선택
       대상: SharePoint(Microsoft Excel 365), Gmail, Anthropic(Claude)
       이유: 인증 토큰은 보안상 자동 삽입이 불가능. 노드당 클릭 2번이면 완료.

  생성 대상: 워크플로 A, B, C, D (4개)

[Phase 2: MCP 서버] (추후 확장)
  n8n REST API에 연결하여 워크플로 생성/수정/실행을 자동화.
  Import 수동 작업 제거 → 완전 자동화.
  이유: 워크플로를 자주 수정/재생성해야 할 때 유용하지만,
  이 프로젝트에서 워크플로는 4개고 한 번 만들면 거의 안 바뀌므로
  Phase 1만으로 충분. Phase 2는 학습/데모 목적으로 추후 진행.

15. 실제 현장 적용 가이드
================================================================================

[동일한 부분 — 변경 없이 재사용]
  ✓ 이상 탐지 로직 (Config-driven, 임계치 튜닝만 필요)
  ✓ AI 해석 (Claude API + 마스킹)
  ✓ 중복 알림 방지
  ✓ 이메일 + 일일 리포트 구조 (AI Agent 포함)
  ✓ Power BI 대시보드 (3페이지)
  ✓ 주간 파일 회전 + 백업
  ✓ 기초 에러 핸들링 (9장: 7개 에러 지점 대응, 데이터 검증, 폴백)

[데이터 소스만 교체]
  입력: data_bank 시뮬레이터 → MES/ERP → SharePoint 자동 업로드
  알림: Gmail → Outlook/Teams (E3에 포함)

[실무에서 추가로 필요한 레이어]

  1. 인증 및 권한 관리
     n8n 접근 권한, SharePoint 파일 권한, Power BI 공유 범위,
     API 키 관리 (mask_dict 보안 저장소 분리 포함)

  2. 에러 핸들링 고도화
     ※ 기초 수준(재시도, 폴백, error_log)은 9장에서 구현됨.
     실무에서 추가로 필요한 것:
     - exponential backoff (재시도 간격을 점진적으로 늘림)
     - 워크플로 실패 시 관리자 자동 알림 (현재는 error_log 기록만)
     - 실행 이력 대시보드 (error_log를 Power BI로 시각화)
     - 연속 실패 감지 (3회 연속 같은 에러 → 에스컬레이션)

  3. 멀티 사용자 / 멀티 공장
     공장별/라인그룹별 알림 수신자 분리, 사용자별 대시보드 필터

  4. 네트워크 보안
     VPN 내 n8n 접근 제한, API 통신 암호화

  5. 운영 안정성
     Docker 컨테이너 자동 재시작, 디스크 모니터링, 백업/복구 절차

  6. 스케일링
     라인 수 대폭 증가 시 Excel → DB 전환 검토

  7. AI 판단 검증
     데모에서는 AI 분석을 그대로 저장하지만,
     실무에서는 담당자가 AI 제언을 확인/승인하는 검토 워크플로 필요.
     예: AI 제언 → Teams 메시지 → 담당자 승인/수정 → 최종 기록

  8. 하위 작업반 계층 구현
     line_master에 상위라인ID/계층 스키마는 준비되어 있으나,
     engine.py에 계층 집계 로직(하위 합산 → 상위 판정) 추가 필요

  9. 운영 시간 설정
     현재 08:00~17:00 고정. 실무에서는 교대 근무 등에 맞게 설정 파일로 변경 가능하게.


  "이 프로젝트는 데모용으로 제작했으며, 핵심 비즈니스 로직은
   실무에서 동일하게 사용할 수 있습니다. 기초 에러 핸들링도 포함되어
   있어서, 7개 에러 지점에 대해 재시도/폴백/로그 기록을 구현했습니다.
   다만 실제 현장에 적용하려면 인증 체계, 에러 핸들링 고도화,
   멀티 사용자 지원 등 운영 레이어를 추가해야 합니다."
